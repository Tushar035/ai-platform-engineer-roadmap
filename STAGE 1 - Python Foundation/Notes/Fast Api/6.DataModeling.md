# 📌 1. Pydantic Models Deep Dive

## 🧠 What is Pydantic?

Pydantic is used for:
- Data validation
- Data parsing
- Type enforcement

👉 FastAPI automatically uses it for:
- Request body validation
- Response serialization

---

## ✅ Basic Model

```python
from pydantic import BaseModel

class User(BaseModel):
    name: str
    age: int
```
💡 Explanation:

Ensures name is string

Ensures age is integer

Automatically validates incoming JSON



---

🔄 Default Values

class User(BaseModel):
    name: str
    is_active: bool = True

💡 Explanation:

If client does not send is_active, default is used



---

🎯 Optional Fields

from typing import Optional

class User(BaseModel):
    name: str
    email: Optional[str] = None

💡 Explanation:

Field may or may not exist

Prevents validation errors



---

🔧 Field Customization

from pydantic import Field

class Product(BaseModel):
    name: str = Field(..., min_length=3)
    price: float = Field(gt=0)

💡 Explanation:

Adds validation rules

Improves API documentation



---

🔁 Model Methods

user = User(name="Tushar", age=25)

user.dict()
user.json()

💡 Explanation:

.dict() → Python dictionary

.json() → JSON string



---

🔄 Model Config

class User(BaseModel):
    name: str

    class Config:
        orm_mode = True

💡 Explanation:

Allows compatibility with ORM (SQLAlchemy)

📌 2. Nested Models & Complex Data


---

🧱 Nested Models

class Address(BaseModel):
    city: str
    zip_code: str

class User(BaseModel):
    name: str
    address: Address

💡 Explanation:

Models inside models

Automatic deep validation



---

📦 List of Models

from typing import List

class Item(BaseModel):
    name: str

class Order(BaseModel):
    items: List[Item]


---

🧠 Deeply Nested Example

class Product(BaseModel):
    name: str
    price: float

class Order(BaseModel):
    products: list[Product]
    total: float


---

🔄 Dictionary Types

from typing import Dict

class ConfigModel(BaseModel):
    settings: Dict[str, str]

💡 Explanation:

Dynamic key-value structure



---

🔀 Union Types

from typing import Union

class ResponseModel(BaseModel):
    data: Union[str, int]

💡 Explanation:

Accepts multiple types



---

🔁 Recursive Models

from typing import List

class Category(BaseModel):
    name: str
    subcategories: List["Category"] = []

📌 3. Dataclasses in FastAPI


---

🧠 What are Dataclasses?

Lightweight alternative to Pydantic models.


---

✅ Basic Example

from dataclasses import dataclass

@dataclass
class User:
    name: str
    age: int


---

🔄 Using with FastAPI

@app.post("/user/")
def create_user(user: User):
    return user


---

💡 Explanation

Less overhead than Pydantic

Faster but less powerful validation



---

⚠️ When to Use Dataclasses?

Use when:

No heavy validation needed

Performance is critical



---

🚫 Limitations

No advanced validation

No automatic docs like Pydantic

📌 4. Advanced Data Types


---

🔐 UUID

from uuid import UUID

class Item(BaseModel):
    id: UUID

💡 Explanation:

Ensures valid UUID format

Used for unique identifiers



---

📧 Email

from pydantic import EmailStr

class User(BaseModel):
    email: EmailStr


---

🕒 DateTime

from datetime import datetime

class Event(BaseModel):
    timestamp: datetime

💡 Explanation:

Automatically parses ISO format



---

🌐 URL

from pydantic import HttpUrl

class Website(BaseModel):
    url: HttpUrl


---

🔢 Decimal (Financial Data)

from decimal import Decimal

class Product(BaseModel):
    price: Decimal

💡 Explanation:

More precise than float

Important for money calculations

📁 Any Type

---

🔥 Real-World Example (Everything Combined)

from pydantic import BaseModel, EmailStr
from uuid import UUID
from datetime import datetime
from typing import List

class Address(BaseModel):
    city: str

class User(BaseModel):
    id: UUID
    email: EmailStr
    created_at: datetime
    addresses: List[Address]


---

🔥 Common Mistakes

Not using proper types

Using float instead of Decimal for money

Ignoring nested validation

Overusing Any type



---

🧠 Best Practices

Use Pydantic models everywhere

Use strong types (UUID, Email, Decimal)

Keep models clean and reusable

Avoid deeply nested complexity (split models)
